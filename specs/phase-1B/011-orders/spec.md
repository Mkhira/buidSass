## Clarifications

### Session 2026-04-22

- Q1: State models (Principle 17 says order, payment, fulfillment, refund are separate) → **A: Four independent state machines**, each with their own column + transition tables: `order_state`, `payment_state`, `fulfillment_state`, `refund_state`. A derived `high_level_status` view exists for storefront UX.
- Q2: Quotation → order relationship? → **A: Quotations are a sibling aggregate** (`quotations` table) that can convert to an order via a dedicated flow; quote price explanation (spec 007-a) is carried over.
- Q3: Order number format? → **A: Human-readable prefixed id**: `ORD-{market}-{yyyymm}-{seq6}` (e.g. `ORD-KSA-202604-000187`), generated by a DB sequence per market+month.
- Q4: Line-level tracking? → **A: Yes.** `order_lines` snapshot the catalog + pricing at time of order; fulfillment + refund can act at line level.
- Q5: Shipments? → **A: One order may produce N shipments.** `shipments` + `shipment_lines` tables; each shipment has its own tracking + state.

## User Scenarios & Testing

### User Story 1 — Customer views their orders (P1)
After checkout, the customer sees the order in their history with state, timeline, items, and actions (reorder, cancel if eligible, download invoice).

**Acceptance Scenarios**:
1. *Given* a confirmed order, *when* the customer opens order history, *then* the order appears with `orderNumber`, `placedAt`, `grandTotal`, `highLevelStatus`.
2. *Given* the order is `placed` and `payment.captured` and no shipment yet, *when* the detail is opened, *then* a "Cancel" action is shown (per cancellation policy).
3. *Given* an order with a captured shipment, *then* tracking link + carrier + ETA are shown.

---

### User Story 2 — Cancel an eligible order (P1)
Customer cancels a just-placed order. Reservations release, payment voided (or refund issued if already captured), order state → `cancelled`.

**Acceptance Scenarios**:
1. *Given* `order.state=placed`, `payment.state=authorized`, no shipment, *when* cancel fires, *then* order → `cancelled`, payment → `voided`, reservations release.
2. *Given* `payment.state=captured`, *then* refund flow (spec 013) is initiated; order → `cancellation_pending` until refund completes, then `cancelled`.
3. *Given* a shipment already exists, *then* cancellation is rejected with `order.cancel.shipment_exists`; customer may file a return (spec 013).

---

### User Story 3 — Fulfillment flow (P1)
Admin marks the order `picking → packed → handed_to_carrier`. A shipment row is created; tracking number is recorded.

**Acceptance Scenarios**:
1. *Given* an eligible order, *when* admin transitions `picking`, *then* `fulfillment.state=picking` and inventory batches are locked at line level.
2. *Given* `packed`, *when* admin selects shipping provider + creates shipment, *then* a shipment row is created with `carrier`, `tracking_number`, `eta`.
3. *Given* `handed_to_carrier`, *then* tracking webhook updates become the source of truth; admin can still override with an audit row.

---

### User Story 4 — Payment webhook advances payment state (P1)
An asynchronous provider sends a `captured` webhook 30 s after submit.

**Acceptance Scenarios**:
1. *Given* order `payment.state=authorized`, *when* webhook `captured` received, *then* state → `captured`; customer + finance notified.
2. *Given* payment fails post-authorization, *then* state → `failed`; order enters `payment_failed_recovery` window (see Principle 17).
3. *Given* duplicate webhook delivery, *then* dedup prevents double state transitions (SC verified).

---

### User Story 5 — Quotation → order conversion (P2)
A B2B buyer accepts a quotation via link. System creates an order at the stored price with no re-pricing.

**Acceptance Scenarios**:
1. *Given* an `active` quotation, *when* accepted, *then* an order is created at the stored `price_explanations` hash.
2. *Given* the stored hash cannot be re-verified (data integrity failure) — `500 order.quote.integrity_fail` + ops alert.
3. *Given* inventory is insufficient at accept time, *then* order is still created but `fulfillment.state=awaiting_stock` with an alert.

---

### User Story 6 — Reorder from history (P2)
Customer taps "Reorder" on a past order. The system seeds a new cart with eligible lines.

**Acceptance Scenarios**:
1. *Given* a past order, *when* reorder fires, *then* a cart is populated with the same products + qtys; unavailable products are skipped and listed in the response.
2. *Given* a restricted product is in the past order, *then* it is added to the cart (Principle 8) with a flag surfaced.

---

### User Story 7 — Admin fulfillment + finance dashboards (P2)
Admin can list orders by state, export for finance, and open an order to see full audit trail.

**Acceptance Scenarios**:
1. *Given* admin list, *when* filters applied (state, market, date range), *then* paginated results return in < 1 s.
2. *Given* an order detail, *then* tabs for Items, Shipments, Payments, Refunds, Audit are visible.
3. *Given* a finance export, *then* CSV download with line-level tax + discount totals (aligns with spec 012).

---

### User Story 8 — Return/refund entry point (P2 seam)
Spec 013 owns returns. Orders expose the return eligibility window + endpoint hooks.

**Acceptance Scenarios**:
1. *Given* an order delivered ≤ 14 days ago (configurable per market), *when* return eligibility is queried, *then* `eligible=true`.
2. *Given* > 14 days, *then* `eligible=false` with `returnWindow.expired`.

---

### Edge Cases
1. Payment state lags fulfillment state (e.g., capture webhook delayed) → admin may manually capture via audit endpoint.
2. Multi-shipment order: each ship advances its own shipment.state; overall `fulfillment.state` derives from aggregate.
3. Item-level cancellation before shipment (admin-only, rare) → `order_lines` flagged `cancelled`; refund computed pro-rata.
4. Duplicate webhook on different orders → dedup per `(provider, event_id)` still applies (shared with spec 010).
5. Order emitted but downstream (search analytics) fails to observe → events retained in outbox for 7 days.
6. Tax rate changed between order and refund → refund uses ORIGINAL rate (stored in explanation).
7. B2B buyer requires invoice before shipment → `payment.state=pending` → shipment allowed after admin unlock (policy-configurable).
8. Quotation conversion races with inventory (someone else bought the stock) → `fulfillment.state=awaiting_stock`, admin notified.
9. Unpaid COD order → `payment.state=pending_cod` until delivery confirmation → `captured`; delivery failure → `failed`.
10. Partial refund after partial return → `refund.state=partial`; order `high_level_status=partially_refunded`.

## Requirements (FR-)
- **FR-001**: System MUST create an order atomically from a confirmed checkout session, carrying over pricing explanation id (spec 007-a).
- **FR-002**: System MUST assign an `order_number` via sequence `ORD-{market}-{yyyymm}-{seq6}`.
- **FR-003**: System MUST maintain four independent state machines (order, payment, fulfillment, refund) with explicit transitions.
- **FR-004**: System MUST expose `POST /v1/customer/orders/{id}/cancel` with policy enforcement (no shipment yet, or refund if captured).
- **FR-005**: System MUST expose admin fulfillment endpoints: advance picking/packed/handed_to_carrier/delivered.
- **FR-006**: System MUST create `shipments` rows when admin triggers a ship; a single order may have N shipments.
- **FR-007**: System MUST accept payment state updates via webhook (shared endpoint with spec 010) and advance `payment_state` accordingly.
- **FR-008**: System MUST allow admin to force payment state transitions with mandatory reason; audit row required.
- **FR-009**: System MUST expose customer read endpoints: list, detail, reorder, return-eligibility, invoice-pdf-link.
- **FR-010**: System MUST expose admin read endpoints: list/filter, detail with all states, audit trail, finance export (CSV).
- **FR-011**: System MUST expose a `quotations` sub-resource: create, list, accept, reject, expire, convert-to-order.
- **FR-012**: Quotation acceptance MUST re-use stored price explanation hash (spec 007-a `issue` mode disabled on accept).
- **FR-013**: System MUST emit events on every state transition (Principle 17): `order.placed`, `order.cancelled`, `payment.captured`, `payment.failed`, `fulfillment.shipped`, `fulfillment.delivered`, etc.
- **FR-014**: System MUST integrate with spec 008 to convert reservations on order creation and to post return movements on refund.
- **FR-015**: System MUST integrate with spec 012 to trigger invoice issuance on `payment.captured`.
- **FR-016**: System MUST integrate with spec 013 as a seam for refund state transitions.
- **FR-017**: System MUST retain order data for 7 years (compliance); soft-delete not permitted on orders.
- **FR-018**: System MUST expose a `high_level_status` computed from the four state machines for simple UX.
- **FR-019**: Every admin mutation MUST write an audit row with actor + before/after JSON (Principle 25).
- **FR-020**: Customer MUST only access their own orders (enforced at handler); B2B account members see company orders per role (spec 004).
- **FR-021**: Reorder MUST produce a new cart, never mutate the original order.
- **FR-022**: Cancellation policy MUST be configurable per market: `hours_after_placed_if_authorized`, `hours_after_captured_if_not_shipped`.
- **FR-023**: System MUST expose `GET /v1/admin/orders/{id}/audit` returning the full audit trail (all four states + who/when).
- **FR-024**: Webhook endpoint MUST be deduped via `(provider_id, event_id)` (shared table with spec 010).
- **FR-025**: Bank-transfer flow MUST allow `AdminConfirmBankTransfer` to advance `payment.state=captured` with an audit row.
- **FR-026**: COD delivery MUST advance `payment.state=captured` when carrier webhook / admin confirms delivery.

### Key Entities
- **Order** / **OrderLine** — the aggregate.
- **Shipment** / **ShipmentLine** — fulfillment tracking.
- **Payment** / **PaymentAttempt** (cross-linked with spec 010).
- **Refund** / **RefundLine** (spec 013 owns deeper semantics; stub here).
- **Quotation** / **QuotationLine**.
- **OrderStateTransition** — audit per-state.

## Success Criteria (SC-)
- **SC-001**: Order creation p95 ≤ 300 ms from checkout `confirm` call.
- **SC-002**: Order number uniqueness: 10 000 concurrent orders across markets → 0 collisions.
- **SC-003**: State-machine invariant: fuzz 100k random transitions → 0 illegal transitions accepted.
- **SC-004**: Cancellation correctness: parameterized tests covering `(order.state × payment.state × shipment_exists)` → 100 % correct policy applied.
- **SC-005**: Webhook dedup: 100 duplicate deliveries → 1 payment-state mutation.
- **SC-006**: Quotation conversion preserves explanation hash byte-identically (SC test).
- **SC-007**: Finance export produces line-level tax+discount matching spec 012 invoice totals for 1000 sampled orders.
- **SC-008**: COD delivery webhook transitions `payment.state` → `captured` within 30 s of receipt.
- **SC-009**: Return eligibility window enforced per market policy (parameterized test).
- **SC-010**: Every admin state mutation leaves an audit row (Principle 25).

## Dependencies
- Spec 004 identity/RBAC + addresses.
- Spec 007-a pricing (explanation hash reuse).
- Spec 008 inventory (convert reservations + returns).
- Spec 010 checkout (creator).
- Spec 012 tax invoices (trigger on capture).
- Spec 013 returns/refunds.
- Spec 014/015 (admin + customer UIs) consume these APIs in Phase 1C.

## Assumptions
- Single currency per order (inherited from market).
- Refunds go to the original payment method by default (spec 013 owns exceptions).
- Bank transfer reconciliation is manual at launch.

## Out of Scope
- Gift cards / store credit — Phase 1.5.
- Order splitting across warehouses at create time (shipments already support N; create-time routing Phase 1.5).
- Subscription orders — Phase 2.
- Marketplace commission math — Phase 2.

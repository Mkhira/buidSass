# Admin Content-Security-Policy

Single source of truth for the CSP, HSTS, and related security headers the admin app emits. Per spec 015 FR-028c.

## Default policy (every admin route)

Emitted by `apps/admin_web/middleware.ts` on every response inside the `(admin)` and `(auth)` groups:

```
Content-Security-Policy:
  default-src 'self';
  script-src 'self' 'nonce-<RANDOM>';
  style-src 'self' 'unsafe-inline';
  img-src 'self' data: https://<storage-host>;
  font-src 'self' data:;
  connect-src 'self' <backend-host> <sse-host> <storage-host>;
  frame-ancestors 'none';
  base-uri 'self';
  form-action 'self';
  object-src 'none';
  upgrade-insecure-requests;

Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
Referrer-Policy: strict-origin-when-cross-origin
X-Content-Type-Options: nosniff
Permissions-Policy: camera=(self), microphone=(), geolocation=()
```

`<RANDOM>` is a per-request 128-bit base64 nonce generated by middleware and propagated to Server Components via `headers()`; Client Components inherit through Next.js's nonce-passing for streamed scripts.

`<backend-host>`, `<sse-host>`, `<storage-host>` are env-driven (`BACKEND_URL`, `NOTIFICATIONS_SSE_URL`, `STORAGE_BASE_URL`).

`Permissions-Policy` reserves `camera=(self)` for spec 017's barcode scanner; everything else is denied.

## Per-route carve-outs

A module that needs a tighter or looser policy on a specific route group declares it here and applies it via `app/(admin)/<feature>/layout.tsx` setting `headers()` for that subtree.

| Route group | Delta | Reason |
|---|---|---|
| `app/(admin)/catalog/products/[productId]` | `script-src` adds `'unsafe-eval'` for `@tiptap/core`'s schema parser | Tiptap's allow-list-only configuration still uses `Function` constructor in one schema-init code path; we accept the carve-out scoped to the editor route only. Reviewed 2026-04-27. |
| `app/(admin)/inventory/adjust` | `Permissions-Policy: camera=(self)` already in default; no delta | Documents the dependency on the global rule; future reviewers see the linkage. |
| (none for 014 customer app) | n/a | Customer app's CSP is owned by the Phase 1E web infrastructure spec. |

Adding a new carve-out requires:

1. A row in this table with the delta and the reason.
2. A code review on the `layout.tsx` change.
3. A re-run of the security-review skill output attached to the PR.

## Reporting (deferred)

CSP violation reporting (`report-to` / `report-uri`) is **not** wired in v1 — observability spec lands the reporting endpoint. Until then, browsers log violations to the console; engineering spot-checks during dev. A Phase 1E task wires `report-to` to a Sentry-equivalent sink.

## Testing

- A Playwright test under `tests/security/csp.spec.ts` walks every admin route and asserts the CSP header is present, the nonce changes per request, and `frame-ancestors 'none'` is enforced (negative test: a `<iframe>` from `evil.test` to the admin app fails to embed).
- A unit test under `tests/unit/middleware/csp.test.ts` asserts the per-route carve-out machinery applies the catalog editor's tightening exactly to its subtree.

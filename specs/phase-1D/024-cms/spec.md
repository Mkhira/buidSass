## Clarifications

### Session 2026-04-29

- Q: Bilingual authoring model — must editors author both `ar` and `en` for every entity, or single-locale with fallback? → A: **Both `ar` and `en` are mandatory for every published content row in the four entity kinds that face the customer in both locales (banner, featured section, FAQ entry, legal page).** A draft MAY exist with one locale only, but the publish action MUST reject with `400 cms.publish.locale_completeness_missing` until both locales carry an editorially-reviewed body. Blog articles MAY be published in a single locale (Principle 4 forbids machine-translation of long-form copy; English-only or Arabic-only blog posts are valid as long as the article carries an explicit `authored_locale` and the customer-side language switcher does NOT auto-translate; the inactive locale renders a "this article is not available in {locale}" placeholder linking to the available-locale version). System-generated chrome (state labels, category labels, breach badges) is fully bilingual via ICU keys regardless of authored locale. Single-locale legal pages are explicitly forbidden — every published legal page version MUST carry both `ar` and `en` bodies.
- Q: Legal page version retention — bounded history, indefinite, or super-admin pruning? → A: **Indefinite retention; legal page versions are append-only and MUST NEVER be hard-deleted from the operational store.** A new published version supersedes the prior version (only the latest published version is `live`); prior versions retain `superseded_at` + `superseded_by_version_id` and remain queryable via the audit-trail / version-history endpoint. Any `DELETE` API call against a legal page version MUST return `405 cms.legal_page.version.delete_forbidden`. Pruning of old versions, even by `super_admin`, is forbidden because legal contracts in force at order time MUST be re-derivable for dispute / refund investigation. Banner slots, featured sections, FAQ entries, and blog articles also MUST NEVER be hard-deleted (FR-005a-style preservation); the "removal" path is `archived` (terminal soft state). The only exception is unsaved drafts — an explicit `DELETE` on a draft (state `draft`, never published) is permitted and audited.
- Q: Preview-token sharing model — open URL, signed token with TTL, or RBAC-only? → A: **Signed preview tokens with bounded TTL; not open URLs and not RBAC-only.** A `cms.editor` or `cms.publisher` MAY mint a preview token for a draft entity via `POST /admin/cms/{kind}/{id}/preview-token`; the token is a signed opaque string with default TTL 24 h (range 1 h – 7 d, configurable per environment) carrying the entity kind, entity id, version id, and a non-PII `actor_role_at_mint` claim. The token grants read-only access to the customer-facing render of the draft via `GET /storefront/cms/preview/{kind}/{id}?token=...` (ungated by sign-in to allow stakeholder review). Preview rendering uses the same template path as published content but stamps a "DRAFT — not for production" banner on the response and excludes the entity from search-engine indexing (`X-Robots-Tag: noindex, nofollow`). Tokens are revocable by minter / `super_admin` via `DELETE /admin/cms/preview-token/{token_hash}`; revocation is immediate (token store is checked on every preview read). Token-store rows are retained 30 days post-expiry for audit, then deleted by a daily worker (the only exception to the "no hard delete" rule, since preview tokens are operational ephemera, not content).
- Q: Featured-section reference resolution — eager copy at publish, or live read? → A: **Live read at storefront request time, with a cached resolved-snapshot for queue display in the admin authoring UI.** A featured section persists only `referenced_entity_kind` + `referenced_entity_id[]` (e.g., `product`, `category`, `bundle`) — never product names, prices, or images. At storefront read time, the storefront read endpoint resolves each reference via the owning module's read contract (`ICatalogProductReadContract`, `ICatalogCategoryReadContract`, `ICatalogBundleReadContract` — declared in `Modules/Shared/`) and merges the live snapshot. Broken references (product archived, category soft-deleted, bundle deactivated) are filtered out of the customer response with a `linked_entity_unavailable` log entry; the section continues to render with the surviving entries. If ALL references resolve unavailable, the section returns `omitted_due_to_unavailable_references=true` to the storefront and an admin alert event `cms.featured_section.fully_broken` is emitted for editor attention. The admin authoring UI shows a per-row "live preview" badge indicating each reference's current availability; references that have gone stale since the section was last edited are flagged with an editor warning but do NOT block save (the section may publish with N - K live references where K is the broken count, as long as N - K ≥ 1).
- Q: Cross-market content-replication guarantees — strict isolation, replication tooling, or cascade? → A: **Strict per-market isolation at V1; explicit "duplicate-to-market" is a deliberate single-action manual flow only for editor convenience.** Each banner / featured section / FAQ entry / legal page version / blog article carries a single `market_code` (one of `EG`, `KSA`, or the literal `*` for cross-market content). Edits to an EG-scoped entity MUST NOT auto-cascade to KSA. An editor MAY trigger a manual "duplicate to market" action that copies the source row's `ar` + `en` bodies + metadata into a new draft scoped to the target market; the duplicate is a fresh draft (state `draft`), is NOT auto-published, requires an editor to review market-specific differences (legal page tax language, payment provider lists, COD-availability text, etc.), and is audited. Cross-market `*` is reserved for the universal-content cases (e.g., a global "we use cookies" banner): when an entity carries `market_code='*'` it is read by every market's storefront; only `super_admin` MAY publish a `*`-scoped legal page version (legal team + super admin pre-condition); `cms.editor` and `cms.publisher` MAY publish `*`-scoped banners / featured sections / FAQ / blog. Customer storefront resolution at read time prefers the most-specific `(market_code, locale)` row and falls back to `*` when no per-market row exists for that locale — never silently substitutes the other market's content.
- Q: Multiple live banners per slot — single-active, multi-active carousel, unbounded, or per-slot variant rule? → A: **Up to N=5 `live` banners MAY co-exist per `(slot_kind, market_code, locale)` simultaneously; the storefront returns them as an ordered list sorted by `priority_within_slot ASC` (with `created_at_utc ASC` as the deterministic tie-break), enabling carousel / rotation rendering on the customer surface.** N=5 is the V1 default and is configurable per market via a `CmsMarketSchema` policy row (`banner_max_live_per_slot`, range 1–10). Publishing a banner that would push the live-count for `(slot_kind, market_code, locale)` past the cap MUST reject with `400 cms.banner.slot_capacity_exceeded`; the editor must archive an existing `live` banner in that slot first OR wait for an in-flight scheduled-end transition to free a slot. Cross-market `*` banners consume one capacity slot in EVERY per-market storefront they participate in (i.e., a `*`-scoped `live` banner with `slot_kind=hero_top` reduces effective capacity in EG by 1 AND in KSA by 1). The customer storefront treats the response as a list — carousel / rotation behavior on the customer surface is owned by spec 014; 024 only guarantees the deterministic ordering and the cap.
- Q: Asset orphan cleanup policy — indefinite retention, immediate deletion, reference-counted sweep, manual-only, or defer to privacy-ops? → A: **Reference-counted retention with a 7-day delayed sweep run by a daily `CmsAssetGarbageCollectorWorker`.** When a draft is hard-deleted or any entity transitions to `archived` / `superseded`, 024 emits a `cms.asset.dereferenced` event for each `asset_id` carried on the affected row. The worker runs daily and, for each dereferenced asset, recounts references across ALL CMS entities in any state (live, draft, scheduled, archived, superseded). Assets with zero remaining references AND ≥ 7 days past the most-recent dereferencing event are deleted via spec 015's storage abstraction; the `CmsAsset` metadata row is retained with `storage_object_state=swept` (FR-005a-style metadata preservation) so audit trails referencing the asset remain resolvable. The 7-day grace window is the recoverability buffer for ops errors (an editor accidentally archives a banner; restores it; the asset is still recoverable). The grace window is configurable per market via `CmsMarketSchema.asset_grace_period_days` (range 0–30; `0` means immediate deletion at the next worker tick). Privacy-ops MAY shorten the window per market when a privacy-incident response demands it. The sweep is audited (one audit row per swept asset). Assets NEVER swept: any asset still referenced by at least one CMS entity row, regardless of state. This is the only hard-delete path for storage objects in 024 (parallel to the preview-token retention sweeper).
- Q: Cross-market `*` ordering vs. specific-market in storefront responses — interleaved by `priority_within_slot` only, specific-market-first, `*`-first, market-only-with-`*`-fallback-when-empty, or per-market-policy? → A: **Specific-market rows always rank ahead of `*`-scoped rows; within each tier, ordering is `priority_within_slot ASC` (banners) / `display_priority ASC` (featured sections) / `display_order ASC` (FAQ) / `published_at_utc DESC` (blog articles), with `created_at_utc ASC` as the deterministic tie-break.** Concretely, for a request `?market=EG&locale=ar`, the response is composed as: first, all rows where `market_code='EG'` ordered by the per-kind sort key; second, all rows where `market_code='*'` ordered by the same key. The capacity cap (FR-021a) applies across the combined response — a `*`-scoped banner consumes 1 capacity slot in EG even though it ranks below all EG-scoped banners; the cap is enforced at publish-time (FR-021a) regardless of tier. This rule applies uniformly to all five storefront read endpoints (banners, featured sections, FAQ, blog articles, legal pages) for consistency: a market-specific row is always the most-specific intent the editor expressed and MUST outrank a global fallback. The `legal_page` storefront read remains exactly one row per `(legal_page_kind, market_code)` — when a market-specific `live` legal page version exists it is returned; when none exists, the `*`-scoped version is returned (the "fallback" semantic is the same single-row substitution rule). Search-engine indexing for `*`-scoped blog articles uses canonical URLs scoped to each market (e.g., `/eg/blog/...` and `/sa/blog/...` both render the same `*`-scoped article body) — duplicate-content concerns are mitigated via `<link rel="canonical">` chosen by spec 014's storefront layer (out of 024's scope).
- Q: Banner CTA reference-staleness — accept-and-render-broken, live-resolve-and-filter, or eager-archive-on-deactivation? → A: **Same live-resolve pattern as featured sections, enforced at BOTH publish-time and storefront-read-time.** At publish-time, when `cta_kind ∈ {product, category, bundle}`, 024 MUST call the corresponding `Modules/Shared/` read contract (`ICatalogProductReadContract`, `ICatalogCategoryReadContract`, `ICatalogBundleReadContract`) to verify `cta_target` is currently resolvable in the entity's `market_code`; if unresolvable, the publish action MUST reject with `400 cms.banner.cta_target_unresolvable`. At storefront-read-time, the banner-list endpoint MUST re-validate every `cta_kind ∈ {product, category, bundle}` reference; banners whose CTA resolves to `linked_entity_unavailable` MUST be silently filtered out of the customer response (treated identically to a fully-broken featured section), an admin alert event `cms.banner.cta_target_broken` MUST be emitted (rate-limited to once per banner per hour to avoid alert storms during catalog maintenance), and the banner row's `cta_health` indicator surfaces the broken state in the admin authoring UI. `cta_kind ∈ {link, external_url, none}` MUST skip resolution (link / external_url validation is a syntactic check at save-time only — `link` MUST be a valid relative path; `external_url` MUST be a valid absolute URL with `https://` scheme). When the catalog read contract returns transient errors (timeout, 5xx), the storefront read MUST fail-open and INCLUDE the banner with a `cta_health=transient_unverified` flag rather than filter it out, since suppressing customer-facing marketing during a catalog blip is worse than a possibly-stale CTA — the storefront itself can decide whether to suppress at render time. The capacity cap (FR-021a) ignores filtered banners (they continue to count against the cap until the editor manually intervenes — the cap measures publish-pressure, not delivered-banner-count, to prevent capacity runaway during catalog churn).
- Q: Stale draft policy — auto-archive after N days, soft-warning with editor-acknowledgement, ownership-reassignment on offboarding, hard-purge after offboarding+grace, or no automation? → A: **Soft-warning queue with editor-acknowledgement; NO auto-archive of any draft.** Drafts older than `CmsMarketSchema.draft_staleness_alert_days` (V1 default 30; range 7–365) MUST be flagged with a `stale=true` indicator on every admin-queue read AND included in the FR-034 metrics endpoint count. A daily `CmsStaleDraftAlertWorker` MUST emit `cms.draft.stale_alert` events (rate-limited to one event per draft per week) for each stale draft; spec 025 subscribers route the digest to the draft's owner + the on-shift `cms.publisher`. NO automatic state transition is performed — drafts ONLY change state via explicit editor action (`editor_save`, `publisher_schedule`, `publisher_publish_now`, `publisher_archive`, `editor_delete_unpublished_draft`) per FR-002. The constitutional principle (Principle 25 — audit + traceability + editorial-grade content) means content lifecycle MUST be editor-owned; auto-archive of a slow-moving legal-page draft, a long-research blog article, or an embargoed-launch banner draft would constitute data loss without consent. Editors MAY explicitly dismiss the stale-alert for a specific draft via `POST /admin/cms/drafts/{id}/dismiss-stale-alert` with `reason_note ≥ 10 chars`; dismissal suppresses the alert for `CmsMarketSchema.draft_staleness_alert_days` days then re-alerts if the draft is still untouched. Drafts owned by an offboarded editor (`cms.editor` role removed from the actor) MUST be auto-flagged `ownership_orphaned=true` on the next worker tick after the role removal event from spec 004 is observed; the orphan flag prompts a `cms.publisher` to re-assign ownership or hard-delete the draft (the FR-005a draft-delete path remains available; this is the only operational on-ramp for clearing genuinely-abandoned drafts).

## Primary outcomes

1. A `cms.editor` can author a banner slot, featured section, FAQ entry, legal page, or blog article in both `ar` and `en` (where dual-locale is mandated), schedule it for a future go-live window, target it to one or more markets, preview it via a signed token before publish, and trust that the publish action records the version, audit row, and storefront-cache invalidation atomically.
2. A `cms.publisher` (and `cms.legal_owner` for legal pages) can take a draft from `draft → scheduled → live` (or directly `draft → live`), and from `live → archived`, with audit on every transition; can never lose history because all prior versions of legal pages and all archived banners / sections / FAQ / articles remain queryable.
3. The customer storefront (spec 014) reads published content via market+locale-aware endpoints with predictable performance (p95 read latency under 200 ms with a backing CDN cache), correct fallback when a locale or market is missing, and zero leakage of `draft` / `scheduled` / `archived` content.
4. Featured sections that reference catalog products, categories, or bundles render correctly even when individual references go stale; broken references are silently filtered out of the customer response and surfaced as editor alerts in the admin queue.
5. Banner slots integrate cleanly with spec 007-b campaigns: a campaign can bind to a banner slot for hero placement, the binding is enforced by a foreign-key-style invariant inside CMS, and lifecycle events (campaign deactivation, banner archival) propagate without leaving orphaned bindings.
6. The data model, market scoping, role boundaries, and content schema are designed so that future multi-vendor expansion (Phase 2) can layer vendor-scoped content authoring on top without rewriting the schema or the lifecycle state machine.
7. Every authored content surface is bilingual AR + EN end-to-end (Principle 4) with editorial-grade copy, full RTL support in the admin authoring UI, and ICU-keyed system chrome.

## Roles and actors

| Role | Permission origin | What they can do in 024 |
|---|---|---|
| `customer` (signed-in or browsing) | spec 004 (or anonymous) | Read published banners, featured sections, FAQ entries, legal pages, blog articles via the storefront read endpoints. No authoring. |
| `cms.editor` | new in this spec | Create + edit drafts of banner slots, featured sections, FAQ entries, blog articles; submit drafts for publish review; mint preview tokens; cannot publish. Cannot author legal page bodies (legal-page authoring requires `cms.legal_owner` or `super_admin`). |
| `cms.publisher` | new in this spec | All `cms.editor` powers plus: publish drafts, schedule publish, archive published content, edit live content (creating a new version), revoke preview tokens. Cannot publish legal page versions. |
| `cms.legal_owner` | new in this spec | Author + publish legal page versions (terms, privacy, returns, cookies) for one or more markets. Cannot author non-legal content unless they are also `cms.editor`. |
| `super_admin` | spec 015 | Implicit superset of all of the above; sole role permitted to publish a `market_code='*'` legal page version. |
| `viewer.finance` | spec 015 | Read-only on legal pages and the legal-page version history (for refund / dispute investigation that requires the contract-in-force at order time). Cannot author or publish. |
| `b2b.account_manager` | spec 021 | Read-only on banners, featured sections, blog articles to align customer outreach with current marketing surface. Cannot author. |

The customer-facing storefront is owned by Phase 1C spec 014 (mobile + web); 024 ships only the backend authoring + storefront read contracts. The admin authoring UI is owned by spec 015 + this spec's contract; 015 builds the screens against 024's endpoints.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — A `cms.editor` schedules a Ramadan campaign banner for KSA, in `ar` and `en`, and publishes it (Priority: P1)

A KSA marketing editor opens the admin CMS module, picks **Banner Slots → Hero Top**, creates a new banner version: uploads a 1920×600 webp asset for `ar` and a separate asset for `en` via the storage abstraction (spec 015 signed-URL flow), authors `headline_ar`, `headline_en`, `subhead_ar`, `subhead_en`, picks `cta_kind=link`, `cta_target=/categories/dental-care/ramadan-offers`, sets `market_code=KSA`, sets `scheduled_start_utc` to the Ramadan-eve midnight in Asia/Riyadh, sets `scheduled_end_utc` to the Eid morning, and saves as `draft`. They mint a preview token and share the resulting URL with the marketing lead. The lead reviews, approves on Slack. The editor returns to the admin UI and submits **Schedule Publish**. The state transitions `draft → scheduled`. At `scheduled_start_utc` the `CmsScheduledPublishWorker` (60-second cadence) finds the row, transitions `scheduled → live`, invalidates the storefront banner cache for `(slot=hero_top, market=KSA, locale=*)`, and emits `cms.banner.published`. At `scheduled_end_utc` the same worker transitions `live → archived` and emits `cms.banner.archived`.

**Why this priority**: Banner slots are the most-frequent CMS surface (Marketing operates them weekly) and the integration with 007-b campaigns depends on banners being live. Without P1, marketing has no path to ship customer-facing promotional surfaces.

**Independent Test**: Author a banner via the admin endpoint with both locales and a future schedule. Verify the row exists in `scheduled`. Manually advance the worker clock to the start time; verify transition to `live` and storefront read returns the banner for KSA + ar. Advance again to the end time; verify transition to `archived` and storefront read no longer returns it.

**Acceptance Scenarios**:

1. *Given* a `cms.editor` posts a draft banner with both `ar` and `en` bodies, two asset ids, `market_code=KSA`, and a future `scheduled_start_utc`, *then* the row persists in `draft` with both bodies and the response echoes the version id + state.
2. *Given* a `cms.editor` attempts to schedule-publish a draft with only `ar` body authored (no `en`), *then* the request rejects with `400 cms.publish.locale_completeness_missing`.
3. *Given* a `cms.publisher` approves the draft, *when* they call schedule-publish, *then* the state transitions `draft → scheduled` and an audit row captures `actor_id`, `from_state=draft`, `to_state=scheduled`, `triggered_by=publisher_schedule`.
4. *Given* a `scheduled` banner reaches `scheduled_start_utc`, *when* the worker tick fires, *then* the state transitions `scheduled → live`, the storefront cache for `(slot=hero_top, market=KSA)` is invalidated, and `cms.banner.published` is emitted.
5. *Given* the same banner reaches `scheduled_end_utc`, *when* the worker tick fires, *then* the state transitions `live → archived` and `cms.banner.archived` is emitted; storefront reads no longer return it.
6. *Given* a `cms.editor` attempts to schedule-publish, *then* the request rejects with `403 cms.publish.role_required`.

### User Story 2 — A customer in EG-AR opens the storefront home and sees the live banner + featured section (Priority: P1)

An anonymous customer in EG opens the customer storefront home with `Accept-Language: ar`. The storefront calls `GET /storefront/cms/banner-slots?market=EG&locale=ar` and `GET /storefront/cms/featured-sections?market=EG&locale=ar`. Both endpoints return only entities in state `live` whose `market_code` matches `EG` or `*` and whose `(scheduled_start_utc <= now() < scheduled_end_utc)` window is open. The featured section endpoint also resolves each `referenced_entity_id` via the catalog read contract; broken references (a product was archived 5 minutes ago) are filtered out of the response with a stable response shape. The customer sees the home banners + featured products in Arabic with RTL layout. A second customer with `Accept-Language: en` (same anonymous IP) gets the EN version of the same entities; the response is the same shape but with `en` bodies.

**Why this priority**: The storefront is the most-read consumer of CMS data; a wrong response shape, a leaked draft, or a stale featured-section reference is a launch-blocker. P1.

**Independent Test**: Seed a `live` banner in EG-AR + EN, a `draft` banner in EG-AR, an `archived` banner in EG-AR, and a `live` banner in `*` cross-market. Call the storefront banner-slot endpoint with `market=EG&locale=ar`; verify only the EG-AR `live` and the `*` `live` rows are returned. Call again with `locale=en`; verify EN bodies. Call with `market=KSA`; verify only the `*` `live` row is returned.

**Acceptance Scenarios**:

1. *Given* a `live` banner in `market_code=EG` and a `live` banner in `market_code='*'`, *when* the storefront calls `GET /storefront/cms/banner-slots?market=EG&locale=ar`, *then* both rows are returned with their `ar` bodies and the response is paged at 50.
2. *Given* a `draft` banner in `market_code=EG`, *when* the storefront calls the same endpoint, *then* the draft is NOT in the response.
3. *Given* a `scheduled` banner whose `scheduled_start_utc` is in the future, *then* the storefront read does NOT return it.
4. *Given* a `live` banner whose `scheduled_end_utc` has passed but the worker has not yet ticked over to `archived`, *then* the storefront read filters it out by `now() < scheduled_end_utc` regardless of state.
5. *Given* a featured section with three product references where one product is now archived, *when* the storefront reads the section, *then* the response includes the section with two resolved-product entries (the archived one filtered out), and an admin alert `cms.featured_section.partial_broken` is emitted (rate-limited to once per hour per section).
6. *Given* the storefront calls with no `Accept-Language`, *then* the endpoint defaults `locale=ar` for `market=EG` / `KSA` (per market default-locale policy from spec 003) and returns the AR bodies.

### User Story 3 — A `cms.legal_owner` publishes a new privacy-policy version after a regulatory change, preserving prior versions (Priority: P1)

The KSA regulatory body issues a new data-residency clause. The legal team drafts a revised privacy policy and hands the markdown bodies (`ar` + `en`) to the `cms.legal_owner`. The owner opens **Legal Pages → Privacy → Edit New Version**, pastes both bodies, sets `effective_at_utc` to the regulator-mandated effective date (10 days out), saves as `draft`. They mint a preview token, share with legal counsel, and after sign-off submit **Schedule Publish**. The state transitions `draft → scheduled`. At `effective_at_utc` the worker transitions to `live`; the prior `live` version transitions to `superseded` (retaining `superseded_at` + `superseded_by_version_id`); both versions remain queryable via the version-history endpoint indefinitely (FR-005a). The storefront cache for `(legal_page=privacy, market=KSA)` is invalidated; subsequent storefront reads return the new version. Customers who consented under the prior version retain their consent record (consent records are owned by spec 004; this spec only tracks which version was live at any UTC moment).

**Why this priority**: Legal pages are the constitutional center of Principle 18 (tax/legal) and Principle 25 (audit). Hard-deleting a legal page version would erase the contract under which prior orders were placed, blocking dispute investigation. P1.

**Independent Test**: Author a new privacy version in `draft`. Verify the prior `live` version is unaffected. Schedule-publish at a future time. Advance the worker clock; verify the prior version transitions to `superseded` and the new version to `live`. Call the version-history endpoint; verify both versions are returned with `effective_at_utc` + `superseded_at` ordering. Attempt `DELETE /admin/cms/legal-pages/{kind}/versions/{id}` against the prior version; verify `405 cms.legal_page.version.delete_forbidden`.

**Acceptance Scenarios**:

1. *Given* a `cms.legal_owner` posts a draft privacy version with both `ar` and `en` bodies and a future `effective_at_utc`, *then* the row persists in `draft`; the prior `live` version is unchanged.
2. *Given* the new version reaches `effective_at_utc`, *when* the worker tick fires, *then* the new version transitions to `live`, the prior version transitions to `superseded` with `superseded_at_utc` and `superseded_by_version_id` populated, and `cms.legal_page.version.published` is emitted.
3. *Given* a customer reads `GET /storefront/cms/legal-pages/privacy?market=KSA&locale=ar` after the transition, *then* the response is the new version's `ar` body.
4. *Given* `viewer.finance` calls `GET /admin/cms/legal-pages/privacy/versions?market=KSA`, *then* the response lists every version (live + superseded + draft + scheduled) with their `effective_at_utc` ordering and the `live` flag.
5. *Given* a `cms.editor` attempts to publish a legal page version, *then* the request rejects with `403 cms.legal_page.publish.role_required` (only `cms.legal_owner` or `super_admin`).
6. *Given* a `cms.legal_owner` attempts to publish a `market_code='*'` legal page version, *then* the request rejects with `403 cms.legal_page.cross_market_requires_super_admin`.
7. *Given* any actor calls `DELETE` on a legal page version, *then* the request rejects with `405 cms.legal_page.version.delete_forbidden`.

### User Story 4 — A `cms.editor` authors and schedules a blog article with SEO fields and previews it via signed token (Priority: P2)

An editor wants to publish a "5 dental hygiene tips" article for the EG market. ### User Story 5 — A `cms.editor` curates a featured section that references three products and one bundle, and one product is later archived in catalog (Priority: P2)

### Edge Cases

- An editor schedules a banner with `scheduled_start_utc > scheduled_end_utc`: rejected with `400 cms.banner.schedule_window_invalid`. The end MUST be strictly after the start; equal is also rejected (zero-duration windows are not permitted).
- An editor publishes a 6th banner into a `(slot_kind, market_code, locale)` that already has 5 `live` banners (V1 default cap from FR-021a): rejected with `400 cms.banner.slot_capacity_exceeded`; the response includes the current `live` banners in that slot to help the editor pick one to archive. A worker `scheduled → live` transition that hits the same cap leaves the row in `scheduled` and emits `cms.banner.scheduled_publish_blocked_capacity` (rate-limited to once per banner per hour) — the editor must intervene.
- An editor schedules a banner with `scheduled_start_utc` in the past: accepted; the worker promotes it to `live` on the next tick (within 60 s) — same behavior as a "publish-now" flow but via the scheduled path. Audit captures `triggered_by=publisher_schedule_past_start` for ops investigation.
- Two editors edit the same draft simultaneously: optimistic-concurrency (xmin row_version). One save succeeds; the other receives `409 cms.draft.version_conflict` with the current version body returned for re-merging in the editor UI.
- A banner asset's storage object is deleted out-of-band (rare ops error): the storefront read returns the banner row with `asset_unavailable_for_locale=true`; admin authoring UI flags the broken asset; editor must re-upload before next save.
- A campaign in spec 007-b binds to a banner slot; an editor archives the banner: the archive request rejects with `409 cms.banner.archive_blocked_by_campaign_binding` and surfaces the bound campaign id; the editor must first unbind in 007-b (or wait for the campaign to deactivate — spec 007-b emits `pricing.campaign.deactivated` events that 024 subscribes to and auto-clears the binding).
- A customer reads the storefront at the exact UTC moment a banner transitions from `scheduled` to `live`: the worker tick boundary is observable via the `now()` filter on `scheduled_start_utc <= now()`. Sub-second consistency is best-effort; eventual consistency within 60 s is the SLA.
- A legal page version is published with `effective_at_utc` in the past: accepted but treated as immediate-effective (not retroactive — the version is `live` from the moment the worker promotes it; it does NOT retroactively apply to prior orders). Audit captures `triggered_by=publisher_schedule_past_effective`.
- A legal page version is scheduled for an `effective_at_utc` that lands inside another scheduled version's window: the second-most-recent `effective_at_utc` wins at the moment its window starts; ordering is strictly by `effective_at_utc ASC`. Editors are warned in the admin UI of overlap before save.
- A featured section references a bundle that gets deactivated (state transition from `active → inactive` in catalog): live storefront read filters it out; treated identically to an archived product per FR-016.
- A blog article's `slug` collides with an existing article in the same `(market_code, authored_locale)`: rejected with `400 cms.blog.slug_collision`; slugs are unique per `(market_code, authored_locale)` tuple.
- An FAQ entry's `display_order` collides with another in the same `(market_code, locale, category)`: accepted (collisions are allowed; ties broken by `created_at_utc ASC`); admin UI surfaces collisions visually for editor cleanup.
- A `cms.editor` mints a preview token then `cms.publisher` revokes it: the next preview-URL request returns `403 cms.preview.token_revoked` immediately (token store is checked on every read).
- A customer storefront read hits a market that has no live banners + no `*` fallback live banners: the response is `{banner_slots: [], market_default_fallback_active: false}`, NOT an error. The customer-app-shell renders an empty hero placeholder.
- A `cms.legal_owner` attempts to publish a legal page version while `super_admin` is concurrently publishing the same legal page kind for the same market: optimistic-concurrency on the legal page row (xmin); one publish wins, the other receives `409 cms.legal_page.version_conflict` with the current state.
- The storefront reads with `market=EG&locale=fr` (unsupported locale): rejected with `400 cms.storefront.locale_unsupported`; supported locales are limited to the spec 003 registry (`ar`, `en` at V1).
- Hard-delete of any non-draft entity is forbidden (FR-005a). The "removal" path is `archived` (terminal soft state). Drafts that have never been published MAY be hard-deleted by their creator or `super_admin` (audited).

## Requirements *(mandatory)*

### Functional Requirements

#### Lifecycle and state model (Principle 24)

- **FR-001**: All five CMS entity kinds — `banner_slot`, `featured_section`, `faq_entry`, `blog_article`, `legal_page_version` — MUST share the same lifecycle state machine: `draft` (initial) → `scheduled` (publish armed for a future `effective_at_utc` / `scheduled_start_utc`) → `live` (visible to storefront) → `archived` (terminal soft-deleted; no longer visible). Direct `draft → live` is permitted (an immediate-publish action behaves as a schedule with `start_utc = now()`). Direct `draft → archived` is permitted (archiving an unpublished draft). `live → archived` is permitted. `archived → *` is forbidden (archived is terminal). For `legal_page_version`, the additional state `superseded` MAY be entered from `live` when a newer version of the same `(legal_page_kind, market_code)` reaches `live`; `superseded` is also terminal.
- **FR-002**: Every state transition MUST write an audit row with `actor_id` (or `'system'`), `actor_role`, `timestamp_utc`, `entity_kind`, `entity_id`, `version_id`, `from_state`, `to_state`, `triggered_by` (one of `editor_save`, `publisher_schedule`, `publisher_publish_now`, `publisher_archive`, `publisher_schedule_past_start`, `publisher_schedule_past_effective`, `worker_promote_to_live`, `worker_promote_to_archived`, `worker_supersede_legal_version`, `super_admin_force`, `editor_delete_unpublished_draft`).
- **FR-003**: Decisions to `live` (via `publisher_publish_now` or `worker_promote_to_live`) MUST require the entity's body / asset state to satisfy the locale-completeness invariant (FR-007); SLA-style override is forbidden.
- **FR-004**: Authoring permissions: `cms.editor` MAY save drafts of `banner_slot`, `featured_section`, `faq_entry`, `blog_article`. `cms.publisher` MAY publish + archive any of those four kinds. `cms.legal_owner` MAY save + publish drafts of `legal_page_version`. `super_admin` MAY do anything. Cross-permission attempts return `403 cms.{entity_kind}.{action}.role_required`.
- **FR-005**: A live entity MUST be readable by the storefront read endpoints; a non-live entity (draft, scheduled, archived, superseded) MUST NOT leak into any storefront read endpoint regardless of caller (auth or anonymous). Verified by a leak-detection test suite.
- **FR-005a**: Banner slots, featured sections, FAQ entries, blog articles, and legal page versions in any post-`draft` state (`scheduled`, `live`, `archived`, `superseded`) MUST NEVER be hard-deleted from the operational store. Any `DELETE` API call against such a row MUST return `405 cms.{entity_kind}.delete_forbidden`. Drafts that have never been published MAY be hard-deleted by their creator or `super_admin` via the explicit `DELETE /admin/cms/{kind}/drafts/{id}` endpoint (audited; rate-limited to 30/hour/actor).

#### Authoring (Principle 4 — bilingual + RTL)

- **FR-006**: All five entity kinds MUST capture content under the per-entity body schema:
  - `banner_slot`: `slot_kind` (one of `hero_top`, `category_strip`, `footer_strip`, `home_secondary`); `headline_ar`, `headline_en`, `subhead_ar`, `subhead_en` (max 120 / 240 chars respectively); `asset_id_ar`, `asset_id_en` (storage abstraction ids per spec 015); `cta_kind` (one of `link`, `category`, `product`, `bundle`, `external_url`, `none`); `cta_target` (depends on `cta_kind`); `scheduled_start_utc`, `scheduled_end_utc`; `market_code` (one of `EG`, `KSA`, `*`); `priority_within_slot` (integer; sort `ASC`; ties broken by `created_at_utc ASC`). Up to `CmsMarketSchema.banner_max_live_per_slot` banners (V1 default 5; range 1–10) MAY be `live` simultaneously per `(slot_kind, market_code, locale)` per FR-021a.
  - `featured_section`: `section_kind` (one of `home_top`, `home_mid`, `category_landing`, `b2b_landing`); `title_ar`, `title_en`, `subtitle_ar`, `subtitle_en`; `references[]` (array of `{kind: 'product'|'category'|'bundle', id: <uuid>}` with min 1, max 24); `market_code`; `display_priority`.
  - `faq_entry`: `category` (one of 8 fixed values: `ordering`, `payment`, `shipping`, `returns`, `account`, `verification`, `b2b`, `other`); `question_ar`, `question_en` (max 250 chars); `answer_ar`, `answer_en` (max 4000 chars; markdown); `display_order` (integer); `market_code`.
  - `blog_article`: `category` (one of 6 fixed values: `tips`, `news`, `guides`, `case_studies`, `clinical`, `other`); `slug` (unique per `(market_code, authored_locale)`; lowercase a-z, 0-9, `-`); `authored_locale` (one of `ar`, `en`); `title`, `summary`, `body` (markdown; max 60000 chars); `cover_asset_id`; `seo` (block with `meta_title`, `meta_description`, `og_image_id`, `schema_org_kind`); `scheduled_publish_at_utc`; `market_code`. Blog articles MAY be authored single-locale (FR-007 exception).
  - `legal_page_version`: `legal_page_kind` (one of 4 fixed values: `terms`, `privacy`, `returns`, `cookies`); `version_label` (free-text; e.g., `2.3.0`); `body_ar`, `body_en` (markdown; both required); `effective_at_utc`; `market_code`. Versioning is per `(legal_page_kind, market_code)` and MUST be strictly ordered by `effective_at_utc ASC`.
- **FR-007**: The publish action (`draft → scheduled` or `draft → live`) MUST enforce locale-completeness:
  - For `banner_slot`, `featured_section`, `faq_entry`: both `ar` and `en` bodies (and both locale assets where applicable) MUST be authored. Single-locale publishes return `400 cms.publish.locale_completeness_missing`.
  - For `legal_page_version`: both `body_ar` and `body_en` MUST be authored AND the `effective_at_utc` MUST be present. Missing either returns `400 cms.publish.locale_completeness_missing` or `400 cms.publish.effective_at_required`.
  - For `blog_article`: only `authored_locale` body MUST be authored; the inactive locale is allowed to be empty (Principle 4 exception per Clarifications). The customer-side storefront response carries `available_locales[]` and a `localization_unavailable_for_requested_locale` flag.
- **FR-008**: The admin authoring UI MUST switch to RTL when the operator's `locale=ar`, including the markdown-editor pane direction, asset-card layout, and form-validation badges. Customer storefront reads return content in the requested `locale` with proper RTL rendering hints (e.g., `dir="rtl"` is emitted on `ar` body responses).
- **FR-009**: Asset uploads MUST go through the spec 015 storage abstraction (signed-URL flow); 024 stores only the `asset_id` on the entity row. Per-asset max size 10 MB (configurable per market; default per asset kind: image 5 MB, video 10 MB). MIME types: `image/png`, `image/jpeg`, `image/webp`, `image/svg+xml`, `application/pdf`, `video/mp4`. Disallowed types return `400 cms.asset.mime_forbidden`.
- **FR-009a**: Asset reference-counting and orphan cleanup MUST be implemented as follows: (a) every entity row carries its `asset_id` references on the row body (banner `asset_id_ar` / `asset_id_en`, blog `cover_asset_id` + embedded markdown asset refs, featured-section asset refs, FAQ markdown asset refs); (b) when a draft is hard-deleted (FR-005a draft-delete path) or any entity transitions to `archived` / `superseded`, 024 MUST emit a `cms.asset.dereferenced` event for each `asset_id` on the affected row, carrying `asset_id`, `entity_kind`, `entity_id`, `dereferenced_at_utc`; (c) a `CmsAssetGarbageCollectorWorker` MUST run on a 24-hour cadence, recounting references across ALL CMS entities in any state (live, draft, scheduled, archived, superseded); (d) assets with zero remaining references AND ≥ `CmsMarketSchema.asset_grace_period_days` days past the most-recent dereferencing event MUST be deleted via spec 015's storage abstraction; (e) the `CmsAsset` metadata row MUST be retained with `storage_object_state=swept` so audit-trail references remain resolvable; (f) every sweep MUST be audited (one audit row per swept asset). The grace window is per-market configurable (V1 default 7 days; range 0–30; `0` permits immediate sweep at the next worker tick). Assets still referenced by at least one CMS entity in any state MUST NEVER be swept regardless of grace age. This is the only hard-delete path for CMS storage objects.

#### Scheduling, publishing, and worker

- **FR-010**: A `CmsScheduledPublishWorker` MUST run on a 60-second cadence, scanning:
  - `scheduled` `banner_slot` rows where `scheduled_start_utc <= now()` and `state = scheduled` → transition to `live`.
  - `live` `banner_slot` rows where `scheduled_end_utc <= now()` and `state = live` → transition to `archived`.
  - `scheduled` `featured_section`, `faq_entry`, `blog_article` rows where `scheduled_publish_at_utc <= now()` → transition to `live`.
  - `scheduled` `legal_page_version` rows where `effective_at_utc <= now()` → transition to `live` AND the prior `live` version (same `(legal_page_kind, market_code)`) → transition to `superseded` with `superseded_at_utc = now()` and `superseded_by_version_id` populated. The two transitions MUST be in a single transaction.
- **FR-011**: Worker transitions MUST be idempotent: re-running the worker on the same row in the same target state MUST NOT emit duplicate domain events or write duplicate audit rows. Idempotency key is `(entity_kind, entity_id, target_state)` per worker tick.
- **FR-012**: The publisher endpoints MUST support both `publish_now` (immediate; transitions `draft → live` if locale-completeness holds) and `schedule_publish` (transitions `draft → scheduled` with the future timestamp). Both paths emit `cms.{entity_kind}.published` on the `live` transition.
- **FR-013**: The archive endpoint (`live → archived`) MUST require an `archive_reason_note ≥ 10 chars` and MUST be performed by `cms.publisher`, `cms.legal_owner` (for legal page versions only), or `super_admin`. Audit captures the reason note.

#### Preview tokens

- **FR-014**: A `cms.editor`, `cms.publisher`, or `cms.legal_owner` MAY mint a preview token for a draft entity via `POST /admin/cms/{kind}/{id}/preview-token` with optional `ttl_hours` (range 1–168; default 24). The response carries the signed opaque token and a copy-able URL `https://storefront/{...}/preview/{kind}/{id}?token=...`.
- **FR-015**: Preview reads (`GET /storefront/cms/preview/{kind}/{id}?token=...`) MUST verify the token signature, check the token store for revocation, render the draft content using the same template path as published content, inject a "DRAFT — not for production" banner marker into the response, and emit `X-Robots-Tag: noindex, nofollow`. Unauthenticated reads are permitted (the token IS the auth).
- **FR-016**: Preview tokens MUST be revocable by the minter, by `cms.publisher` / `cms.legal_owner`, or by `super_admin` via `DELETE /admin/cms/preview-token/{token_hash}`. Revocation is immediate (token store check on every read). A daily worker MUST delete token-store rows ≥ 30 days past TTL expiry; this is the only hard-delete path in the spec (preview tokens are operational ephemera, not content).

#### Storefront reads

- **FR-017**: Storefront read endpoints MUST be public (no auth required) and MUST filter to `state=live`, `(scheduled_start_utc IS NULL OR scheduled_start_utc <= now())`, `(scheduled_end_utc IS NULL OR scheduled_end_utc > now())`, and `(market_code = {requested_market} OR market_code = '*')`.
- **FR-018**: Storefront responses MUST honor the requested `locale` and never auto-translate. For entities with bilingual mandatory content (banner, featured section, FAQ, legal page), the requested locale's body MUST be returned. For blog articles authored single-locale, when the requested `locale` differs from `authored_locale`, the response includes `body=null`, `available_locales=[<authored>]`, `localization_unavailable_for_requested_locale=true`.
- **FR-019**: The storefront featured-section read MUST resolve each `references[]` entry via the corresponding `Modules/Shared/` read contract (`ICatalogProductReadContract`, `ICatalogCategoryReadContract`, `ICatalogBundleReadContract`). Unavailable references are silently filtered out of the response with the response shape `{section_id, references_resolved:[...], total_references, total_resolved, total_unavailable}`. When `total_resolved == 0`, the section is included with `omitted_due_to_unavailable_references=true` and an admin alert event `cms.featured_section.fully_broken` is emitted (rate-limited to once per hour per section).
- **FR-020**: Storefront responses MUST be cache-friendly: response headers include `Cache-Control: public, max-age=60, stale-while-revalidate=300` and a stable `ETag` derived from the response payload hash. CDN cache invalidation on publish / archive is handled by emitting `cms.cache.invalidate.{kind}` events to spec 025's edge-cache subscriber (in V1, the subscriber is a stub; the events are emitted from V1 to unblock edge-cache wiring in Phase 1E).
- **FR-021**: Storefront read endpoints MUST be paged at default 50 per page, max 200, with a two-tier sort: (a) all rows where `market_code` matches the requested market come first; (b) all rows where `market_code='*'` come second. Within each tier, the per-kind sort key is `priority_within_slot ASC` then `created_at_utc ASC` (banners), `display_priority ASC` then `created_at_utc ASC` (featured sections), `display_order ASC` then `created_at_utc ASC` (FAQ), `published_at_utc DESC` (blog articles). Legal pages return exactly one row per `(legal_page_kind, market_code)`: if a market-specific `live` version exists it is returned; otherwise the `*`-scoped `live` version is returned; otherwise the response is a 404. A separate `version-history` endpoint exposes prior `superseded` versions to admin roles.
- **FR-021a**: Banner-slot capacity per `(slot_kind, market_code, locale)` MUST be capped at `CmsMarketSchema.banner_max_live_per_slot` (V1 default 5; range 1–10; configurable per market by `super_admin`). The cap counts banners in state `live` whose `(scheduled_start_utc IS NULL OR scheduled_start_utc <= now())` AND `(scheduled_end_utc IS NULL OR scheduled_end_utc > now())`. Cross-market `*`-scoped banners count against the cap in every per-market view they appear in (i.e., a `live` `*`-scoped `hero_top` banner consumes 1 capacity in `EG` AND 1 in `KSA`). A publish action (`draft → live` or `scheduled → live`) that would push the effective live-count past the cap MUST reject with `400 cms.banner.slot_capacity_exceeded` and surface the current `live` banners in that slot for the editor to triage. The `CmsScheduledPublishWorker` MUST also enforce the cap on `scheduled → live` transitions; if the cap is full at the worker tick (e.g., another banner was published manually after this one was scheduled), the worker leaves the row in `scheduled` and emits `cms.banner.scheduled_publish_blocked_capacity` for editor attention (rate-limited to once per banner per hour). The customer storefront response is the deterministic ordered list — carousel / rotation behavior is owned by spec 014.

#### Cross-module integration

- **FR-022**: Featured section references MUST be resolved via per-module read contracts declared in `Modules/Shared/` to avoid module dependency cycles, consistent with the project's existing pattern (specs 020 / 021 / 022 / 023). 024 MUST NOT take a hard EF Core navigation property to catalog tables; the relationship is logical-only via the polymorphic `references[]` array.
- **FR-022a**: Banner-slot `cta_target` references MUST be validated via the same `Modules/Shared/` read contracts as featured sections, BOTH at publish-time and at storefront-read-time. (a) At publish-time, when `cta_kind ∈ {product, category, bundle}`, 024 MUST call the corresponding read contract to verify the target is resolvable in the banner's `market_code`; unresolvable targets MUST reject with `400 cms.banner.cta_target_unresolvable`. (b) At storefront-read-time, the banner-list endpoint MUST re-validate every catalog-bound CTA; banners whose CTA target is unresolvable (target archived / soft-deleted / market-mismatched after publish) MUST be silently filtered out of the customer response and `cms.banner.cta_target_broken` MUST be emitted (rate-limited to once per banner per hour). (c) When the catalog read contract returns a transient error, the banner MUST be returned with `cta_health=transient_unverified` (fail-open semantic) rather than filtered out. (d) `cta_kind ∈ {link, external_url, none}` MUST skip catalog resolution; `link` is a syntactic relative-path check at save-time, `external_url` is a syntactic absolute-`https://` URL check at save-time, `none` is a no-op. (e) The FR-021a capacity cap counts banners regardless of CTA health (a banner with broken CTA still consumes its slot until the editor archives or fixes it).
- **FR-023**: Banner slots MAY be bound to a spec 007-b campaign for hero placement. The binding is captured in a `BannerCampaignBinding` row carrying `banner_id` + `version_id` + `campaign_id`. 024 subscribes to spec 007-b's `pricing.campaign.deactivated` event and auto-clears the binding (sets `binding_state=released_due_to_campaign_deactivation`). An editor archive request on a banner with an active campaign binding MUST reject with `409 cms.banner.archive_blocked_by_campaign_binding` until the campaign deactivates or an editor manually unbinds in 007-b.
- **FR-024**: 024 MUST emit domain events consumed by spec 025 (notifications), spec 028 (analytics), and spec 014 (storefront edge cache):
  - Lifecycle: `cms.banner.published`, `cms.banner.archived`, `cms.featured_section.published`, `cms.featured_section.archived`, `cms.faq.published`, `cms.faq.archived`, `cms.blog_article.published`, `cms.blog_article.archived`, `cms.legal_page.version.published`, `cms.legal_page.version.superseded`.
  - Operational: `cms.featured_section.partial_broken`, `cms.featured_section.fully_broken`, `cms.cache.invalidate.banner`, `cms.cache.invalidate.featured_section`, `cms.cache.invalidate.faq`, `cms.cache.invalidate.legal_page`, `cms.cache.invalidate.blog_article`, `cms.banner.scheduled_publish_blocked_capacity`, `cms.banner.cta_target_broken`, `cms.asset.dereferenced`, `cms.asset.swept`, `cms.draft.stale_alert`, `cms.draft.ownership_orphaned`.
  - Preview: `cms.preview.token_minted`, `cms.preview.token_revoked`.
- **FR-025**: 024 MUST NOT directly send notifications; it only emits events.

#### Audit (Principle 25)

- **FR-026**: Every state transition, every draft save (create + update), every publisher / legal-owner action, every preview token mint / revoke, every reorder bulk-update, every banner-campaign binding / unbinding MUST emit an audit row to the shared audit log via `IAuditEventPublisher`.
- **FR-027**: Audit rows MUST be immutable and MUST NOT be deletable from any UI; the underlying audit-log table is owned by spec 003 and append-only.

#### SEO and metadata

- **FR-028**: `blog_article` rows MUST carry an `seo` block with `meta_title` (max 70 chars), `meta_description` (max 160 chars), `og_image_id` (storage asset id; optional), `schema_org_kind` (one of `Article`, `BlogPosting`, `NewsArticle`, `FAQPage`); the storefront blog-article read endpoint MUST surface this block at the response root. `featured_section`, `faq_entry`, `banner_slot`, `legal_page_version` MAY carry an `seo` block (optional); when present it is exposed identically.
- **FR-029**: The storefront MUST emit JSON-LD structured data for `legal_page_version` (`schema_org_kind=WebPage` by default) and `blog_article` (per `seo.schema_org_kind`). Generation is the storefront's responsibility (spec 014); 024 only ensures the `seo` block is in the read payload.

#### Multi-vendor readiness (Principle 6)

- **FR-030**: Every CMS entity row MUST carry a `vendor_id` column (nullable in V1; populated only when the entity is vendor-scoped, which is V1-default-NULL for all kinds). Indexed for future-vendor-scoped reads. The admin UI MUST NOT expose vendor scoping in V1; the column is forward-compatibility only.

#### Operational safeguards

- **FR-031**: All admin endpoints MUST require authentication and the corresponding RBAC permission; lookups MUST cap result-set size at 200 with paging.
- **FR-032**: Admin actions MUST be rate-limited per `actor_id`: editor save 60 / minute (defeats automation abuse); publisher publish 30 / hour; archive 30 / hour; preview-token mint 30 / hour. Over-limit returns `429 cms.admin_rate_limit_exceeded`.
- **FR-033**: Draft create / update, publish, archive, and preview-token mint endpoints MUST require an `Idempotency-Key` header (consistent with spec 015 idempotency middleware) so retried network-flaky writes do not duplicate rows.
- **FR-034**: The admin CMS module MUST be observable: per-kind active-content counts (banners live, FAQ entries live, etc.), broken-reference counts (featured sections + banner CTAs), draft staleness (drafts older than `CmsMarketSchema.draft_staleness_alert_days`; V1 default 30), `ownership_orphaned` draft counts (drafts whose owner's `cms.editor` role was revoked per FR-034a), and pending-scheduled counts MUST be exposed via a metrics endpoint readable by `super_admin` and `viewer.finance`. Detailed analytics dashboards are owned by spec 028 (Phase 1.5).
- **FR-034a**: A `CmsStaleDraftAlertWorker` MUST run on a 24-hour cadence to (a) flag drafts whose `created_at_utc` (or last `editor_save_at_utc`) is older than `CmsMarketSchema.draft_staleness_alert_days` with `stale=true` on admin-queue reads; (b) emit `cms.draft.stale_alert` events for each stale draft (rate-limited to once per draft per week, suppressed entirely while a `dismiss-stale-alert` is in-effect); (c) detect drafts whose owner's `cms.editor` role has been revoked (subscribed via spec 004's `customer.role_revoked` channel) and flag `ownership_orphaned=true`. Auto-archive of any draft is FORBIDDEN — drafts only change state through explicit editor / publisher / legal-owner action per FR-002. The dismiss-stale-alert endpoint (`POST /admin/cms/drafts/{id}/dismiss-stale-alert`) requires `reason_note ≥ 10 chars` and is audited; dismissal suppresses re-alerting for `CmsMarketSchema.draft_staleness_alert_days` days then re-alerts if the draft remains untouched. The `ownership_orphaned` flag MUST persist on the draft until a `cms.publisher` either re-assigns ownership (audited) or hard-deletes the draft via the FR-005a draft-delete path.

### Key Entities

- **BannerSlot** — versioned hero / category-strip / footer-strip slot with bilingual text, per-locale assets, market scope, schedule window, optional CTA, and optional 007-b campaign binding.
- **FeaturedSection** — curated cluster of catalog references (products / categories / bundles) on a customer-facing surface; references are id-only with live resolution at storefront read time.
- **FaqEntry** — bilingual question / answer entry with category and per-market display ordering.
- **BlogArticle** — single-locale or bilingual long-form article with markdown body, slug, category, SEO block, and scheduled publish window.
- **LegalPageVersion** — append-only versioned legal contract (terms / privacy / returns / cookies) per `(legal_page_kind, market_code)`; prior versions remain queryable indefinitely.
- **CmsAsset** — link from a CMS entity to a spec 015 storage object id; carries MIME, size, intended-locale tag, original filename.
- **CmsPreviewToken** — signed opaque token granting time-bounded read access to a draft entity via the storefront preview endpoint; revocable; 30-day post-expiry retention then daily-worker deletion.
- **BannerCampaignBinding** — link between a banner slot version and a 007-b campaign; cleared automatically on campaign deactivation; blocks banner archival while active.
- **CmsMarketSchema** — per-market policy row holding capacity + window defaults: `banner_max_live_per_slot` (V1 default 5; range 1–10), `featured_section_max_references` (V1 default 24), `preview_token_default_ttl_hours` (V1 default 24), `draft_staleness_alert_days` (V1 default 30), `asset_grace_period_days` (V1 default 7; range 0–30). Editable by `super_admin` only; edits are audited.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A `cms.editor` MUST be able to author a bilingual banner slot with one asset per locale, schedule it for a future window, and submit it in under 90 seconds from "New Banner" tap to confirmation.
- **SC-002**: 100 % of `state` transitions, draft saves, publisher / legal-owner actions, preview-token mints / revokes, reorder bulk-updates, and banner-campaign binding / unbinding MUST produce a matching audit row, verified by the audit-coverage script in spec 015.
- **SC-003**: 0 % of `draft`, `scheduled`, `archived`, or `superseded` content MAY appear in any storefront read endpoint response, verified by a leak-detection test suite that exercises every storefront read endpoint with fixtures containing all four non-live states.
- **SC-004**: Worker-driven state transitions (scheduled → live, live → archived, scheduled → live with prior superseded for legal pages) MUST occur within 60 seconds of the target UTC moment, p95, measured over a 7-day soak.
- **SC-005**: Worker transitions MUST be idempotent: under a 100-iteration repeat-worker stress test against the same backdated row, exactly 1 transition + 1 audit row + 1 emitted event MUST exist per `(entity_kind, entity_id, target_state)` tuple.
- **SC-006**: The storefront banner-slot read endpoint MUST return a 50-row page in under 200 ms p95 against a seeded fixture of 1,000 live banners, with CDN-friendly cache headers (`Cache-Control` + stable `ETag`) on every response.
- **SC-007**: The storefront featured-section read endpoint MUST resolve a 24-reference section in under 300 ms p95 against a seeded fixture of 10,000 catalog products, including per-reference availability checks via the catalog read contract.
- **SC-008**: AR-locale screen-render correctness (RTL, label completeness, formatting) on the admin authoring UI MUST score 100 % against a representative 30-screen editorial-review checklist (Principle 4).
- **SC-009**: The `cms-v1` seeder MUST populate ≥ 1 row in each of `draft`, `scheduled`, `live`, `archived` states across all five entity kinds, with at least one bilingual + one cross-market `*` example per kind, in under 20 s on a fresh staging DB.
- **SC-010**: Concurrent draft-save races on the same entity MUST resolve with exactly one winning save under a 100-concurrent-attempt stress test (xmin row_version optimistic concurrency).
- **SC-011**: Hard-delete attempts against any non-draft entity MUST return `405 cms.{entity_kind}.delete_forbidden` in 100 % of test cases (FR-005a verification).

## Assumptions

- The 015 `admin-foundation` shell (RBAC, audit panel, idempotency middleware, rate-limit middleware, storage abstraction with signed-URL upload + retrieval) is at DoD on `main` before 024 implementation begins.
- The 003 `markets-and-localization` registry exposes a deterministic resolver for `(market_code, locale)` defaults and supported-locale lists; 024 calls it at storefront read time to validate `locale` parameters.
- The 005 `catalog` module exposes per-kind read contracts (`ICatalogProductReadContract.Read(product_id) → {summary_view, vendor_id, market_code, state}`, similarly for category + bundle) declared in `Modules/Shared/`. If the contracts are not yet on `main` when 024 implementation starts, 024 ships against documented stubs and integration tests assert against fakes.
- The 007-b `promotions-ux-and-campaigns` module emits `pricing.campaign.deactivated` events on the in-process MediatR bus when a campaign moves to a terminal lifecycle state; 024 subscribes via the same channel pattern used by specs 020 / 021 / 022 / 023.
- The 002 `audit-and-events` audit-log table + `IAuditEventPublisher` interface are at DoD; 024 writes audit rows via the shared publisher only — no direct table writes.
- The 025 `notifications` module is implemented in Phase 1E; 024 emits domain events on the existing in-process bus from V1, and 025 subscribers are wired up at 1E DoD. Until 1E ships, edge-cache invalidation events are recorded as "would-have-emitted" log entries — but the events themselves MUST emit from V1 onward to unblock the 1E subscribers without retro-fitting.
- Storage for CMS assets uses the existing spec 015 storage abstraction (signed-URL upload + retrieval); 024 stores only the storage `asset_id` array on the entity row. The CDN sitting in front of storage is owned by Phase 1E E1 (infrastructure-integration); cache-invalidation for CDN edges is best-effort in V1 and TTL-driven (60 s `max-age`).
- Single-vendor at V1 (Principle 6); `vendor_id` columns on every entity row are present and indexed but not exposed in admin UI.
- Operators sign in through the spec 015 admin shell; this spec does not introduce a new auth path.
- Currency / market resolution per market is fixed (EG → EGP, KSA → SAR); CMS content does not embed currency-formatted figures; commerce figures are read by the storefront from the catalog / pricing modules at render time.
- The default V1 locale set is `ar` + `en` only; additional locales (e.g., `fr`, `tr`) are out of scope and any `?locale=fr` storefront read returns `400 cms.storefront.locale_unsupported`.
- The default V1 market set is `EG` + `KSA` + `*` (cross-market) only.
- Featured-section reference resolution is live-read at storefront request time (not eager-copy). The trade-off is one extra round-trip per reference vs. always-fresh data; SC-007's 300 ms p95 budget accounts for the resolution overhead.
- Preview tokens are signed opaque strings (HMAC-SHA256 over the entity kind, id, version, mint timestamp, TTL) using a server-side secret rotated per Phase 1E E1 Key Vault policy. The signing key is NOT stored in `appsettings.json`.
- Legal page version-history is queryable indefinitely; storage growth for legal page versions is bounded operationally (regulatory-driven; expected ≤ 4 versions per `(kind, market)` per year on average) and does not require pruning policy in V1.

## Out of Scope

- **Rich WYSIWYG / drag-and-drop page builder** — Phase 1.5+; V1 ships a markdown editor + structured form fields per entity kind.
- **A/B testing of banner / featured-section variants** — Phase 1.5; V1 publishes one version at a time per `(slot, market_code, locale)`.
- **Personalized content per customer segment** — Phase 1.5; V1 targets only by `market_code` + `locale` + (for banners) `slot_kind` + scheduled window.
- **Localized URLs / per-locale slugs for blog articles** — partial; blog slug uniqueness is per `(market_code, authored_locale)` in V1, but localized URL routing on the storefront is owned by spec 014.
- **Workflow / multi-step approval beyond editor → publisher** — Phase 1.5; V1 has a single approval gate (publisher action).
- **External CMS import / RSS / WordPress migration tooling** — out of scope.
- **Tag-based content discovery / related-articles algorithm** — Phase 1.5; V1 exposes only category-based grouping.
- **Comment threads / customer engagement on blog articles** — Phase 2; blog is read-only at V1.
- **Customer-submitted content (testimonials, case-study submissions)** — Phase 2; all content is editor-authored at V1.
- **Auto-translation between AR and EN** — explicitly rejected per Principle 4; all bilingual content is editorially authored.
- **Region-aware geo-IP fallback for the storefront market resolution** — owned by spec 014 + spec 003; 024 only honors the `?market=` query parameter passed in by the storefront.
- **Vendor-scoped content authoring** — Phase 2 multi-vendor.
- **AI-assisted copy generation / SEO suggestions** — Phase 1.5; editors write copy by hand at V1.
- **Bulk import of FAQ entries / blog articles via CSV / JSON upload** — Phase 1.5; V1 ships per-entity authoring only (the seeder covers the bulk-load V1 path for staging).
- **Content scheduling timezones beyond UTC input** — out of scope; admin UI converts the editor's local timezone into UTC at save (display-only convenience); storage and worker comparisons are UTC-only.
- **Public RSS / Atom feeds for the blog** — Phase 1.5; V1 exposes only the storefront blog-list endpoint to spec 014.
